# Harness Builder 스킬

사용자의 직무, 핵심 업무, 도구, 제약사항을 인터뷰로 수집한 뒤,
Claude Code 하네스와 Cowork 커스텀 스킬 구조 초안을 제시하고,
승인 후 실제 파일을 생성하는 인터뷰어 + 생성기 스킬.

## 트리거

`/harness-builder` 호출 또는 "하네스 만들어줘", "내 Claude Code 환경 설정해줘",
"커스텀 스킬 구조 설계해줘" 등의 자연어 요청.

---

## 전체 흐름 (4단계)

```
Phase 1 (인터뷰)  →  Phase 2 (추가 질문, 필요 시 반복)
      ↓
Phase 3 (초안 제시 + 승인 대기)
      ↓  승인
Phase 4 (파일 생성 + 설치 안내)
```

**핵심 규칙: Phase 3 초안을 사용자가 명시적으로 승인하기 전까지는 어떤 파일도 생성하지 않는다.**

---

## Phase 1 — 기본 인터뷰 (라운드형)

한 번에 한 라운드씩 질문한다. 이전 라운드 답변을 받은 후에만 다음 라운드로 넘어간다.

### 라운드 1 — 역할과 컨텍스트 (3문항)

1. 직함 / 역할이 무엇인가요?
2. 어떤 팀 또는 조직에서 일하나요?
3. 일상 업무에서 가장 중요한 목표는 무엇인가요?

### 라운드 2 — 핵심 업무 (4문항)

1. 가장 자주 하는 업무 3~5가지는 무엇인가요?
2. 그 중 반복적이거나 자동화하고 싶은 업무는 어떤 건가요?
3. Claude가 처음부터 끝까지 대신 처리해줬으면 하는 업무가 있나요?
4. 현재 가장 시간이 많이 걸리는 작업은 무엇인가요?

### 라운드 3 — 도구와 시스템 (4문항)

1. 주로 사용하는 툴, CLI, 서비스는 무엇인가요? (예: GitHub, Jira, Slack, BigQuery, AWS)
2. Claude가 연동해야 할 내부 API, DB, 대시보드가 있나요?
3. 주로 사용하는 언어와 프레임워크는 무엇인가요?
4. 현재 Claude Code에서 사용 중인 MCP 서버가 있나요?

### 라운드 4 — 제약사항과 선호도 (4문항)

1. Claude가 자동으로 하면 절대 안 되는 행동이 있나요? (예: main 브랜치 push, 메시지 전송, 데이터 삭제)
2. 보안 또는 컴플라이언스 제약이 있나요? (예: 외부 네트워크 호출 금지, PII 로깅 금지)
3. Claude가 안전한 범위에서 자율적으로 행동하길 원하나요, 아니면 매번 확인을 받길 원하나요?
4. 승인 없이 자동 실행을 허용할 수 있는 작업 유형이 있다면 말씀해주세요.

### 라운드 5 — 기존 설정 (선택, 3문항)

1. 이미 CLAUDE.md, settings.json, 또는 hooks가 설정되어 있나요?
2. 이전에 자동화를 시도해봤다가 잘 안 됐던 워크플로우가 있나요?
3. 이 하네스를 주로 어떤 프로젝트 디렉토리에서 사용할 예정인가요?

---

## Phase 2 — 충분성 검사 및 추가 질문 (반복 가능)

아래 체크리스트를 모두 충족할 때까지 Phase 3으로 넘어가지 않는다.

**충분성 체크리스트:**

- [ ] 역할과 주요 목표가 명확함
- [ ] 반복 업무 3개 이상이 구체적으로 파악됨
- [ ] 주요 도구/시스템 목록이 있음
- [ ] 최소 1개 이상의 제약사항 또는 안전 경계가 정의됨
- [ ] 자율 실행 vs. 확인 요청 선호도가 파악됨
- [ ] 생성할 스킬의 범위가 2개 이상 도출 가능함

부족한 항목이 있으면 해당 항목에 대한 추가 질문을 1~3개 던진다.
답변이 모호하면 한 번 더 구체적인 예시를 요청한다.
추가 질문은 최대 2라운드까지 반복할 수 있다.

---

## Phase 3 — 초안 제시 및 승인 대기

충분성 기준을 모두 충족하면 아래 형식으로 초안을 제시한다.
**초안 제시 후 반드시 "이 초안대로 파일을 생성할까요?"라고 물어본다.**
사용자가 명시적으로 승인("네", "좋아", "생성해줘" 등)하기 전까지 어떤 파일도 만들지 않는다.
수정 요청이 오면 초안을 업데이트하고 다시 승인을 기다린다.

### 초안 형식

---

### [초안] 생성될 결과물 목록

#### A. Claude Code 하네스

**1. CLAUDE.md**

- 프로젝트 개요, 주요 커맨드, 아키텍처 요약
- 스킬 사용 방법 및 호출 패턴
- 제약사항 및 자동화 범위 명시

**2. .claude/settings.json**

```json
// 예시 구조 (실제 값은 인터뷰 내용 기반으로 채움)
{
  "permissions": {
    "allow": ["<자동 허용 도구 목록>"],
    "deny": ["<금지 도구 목록>"]
  },
  "hooks": {
    "PreToolUse": [{ "matcher": "<조건>", "hooks": [{ "type": "command", "command": "<명령>" }] }],
    "PostToolUse": [...],
    "Stop": [...]
  },
  "env": {
    "<KEY>": "<VALUE>"
  }
}
```

각 hook과 permission에 대해 왜 이 설정이 필요한지 한 줄씩 이유를 설명한다.

**3. .claude/skills/<스킬명>/SKILL.md** (스킬마다 생성)

파일 맨 위에 반드시 YAML frontmatter를 포함한다:

```markdown
---
name: <skill-name>
description: <이 스킬이 언제 쓰이는지 한 줄 설명>
---
```

본문 구조:

```
스킬명: <name>
트리거: /<command> 또는 자연어 패턴
목적: 한 문장으로 기능 설명
입력: 사용자 또는 컨텍스트에서 필요한 정보
단계: Claude가 수행하는 순서
출력: 사용자가 받는 결과물
제약: 절대 하지 않을 행동 (인터뷰 내용 기반)
```

**4. docs/templates/** (필요한 경우)

- 스킬이 사용하는 문서 템플릿 파일 목록 및 위치

---

#### B. Cowork 커스텀 스킬 폴더

업로드 가능한 구조로 생성된다:

```
cowork-skills/
└── <skill-name>/
    ├── skill.json          # 스킬 메타데이터 및 트리거 정의
    ├── prompt.md           # 스킬 시스템 프롬프트
    └── README.md           # 스킬 설명 및 사용법
```

**skill.json 예시 구조:**

```json
{
  "name": "<스킬명>",
  "description": "<한 줄 설명>",
  "trigger": "<트리거 패턴>",
  "version": "1.0.0",
  "constraints": ["<제약사항 목록>"]
}
```

---

#### C. 제안 스킬 목록 (2~5개)

인터뷰 내용을 바탕으로 가장 반복적이고 효과가 높은 업무 순서로 나열:

| 순위 | 스킬명 | 목적 | 트리거 | 우선 이유 |
| ---- | ------ | ---- | ------ | --------- |
| 1    | ...    | ...  | /...   | ...       |

## 기본 생성 규칙 (항상 적용)

아래 규칙은 사용자가 따로 요청하지 않아도 Phase 4 파일 생성 시 항상 적용한다.

1. **SKILL.md frontmatter 필수**: 생성되는 모든 `.claude/skills/<skill-name>/SKILL.md` 파일 맨 위에 반드시 YAML frontmatter를 포함한다. 최소한 `name`과 `description` 필드를 포함해야 한다.
2. **frontmatter 형식**: `name`은 스킬 디렉토리명과 동일하게, `description`은 이 스킬이 언제 쓰이는지 한 줄로 기술한다.
3. **drafts/ 폴더 선행 생성**: 파일 생성 전에 `drafts/` 폴더가 없으면 먼저 생성한다.
4. **cowork-skills/ 폴더 선행 생성**: `cowork-skills/` 및 각 하위 스킬 폴더가 없으면 먼저 생성한다.
5. **모든 필요 디렉토리 선행 확인**: 파일을 쓰기 전에 부모 디렉토리가 존재하는지 확인하고, 없으면 생성한다. Write 실패를 방지하기 위해 Bash `mkdir -p` 로 한 번에 처리한다.
6. **디렉토리 생성은 파일 생성보다 먼저**: 모든 필요한 디렉토리를 한 번의 Bash 명령으로 생성한 뒤, 파일 Write를 진행한다.

## Phase 4 — 파일 생성 및 설치 안내

사용자가 승인하면 아래 순서로 실행한다.

### 생성 순서

0. **디렉토리 선행 생성** (Bash `mkdir -p` 한 번에 처리):
   - `.claude/skills/<스킬명>/` (스킬마다)
   - `drafts/`
   - `cowork-skills/<스킬명>/` (스킬마다)
   - 기타 필요한 모든 경로
1. `CLAUDE.md` 생성 (프로젝트 루트)
2. `.claude/settings.json` 생성 또는 업데이트
3. `.claude/skills/<스킬명>/SKILL.md` 생성 — **반드시 frontmatter 포함** (스킬마다)
4. `docs/templates/` 하위 템플릿 파일 생성 (필요 시)
5. `cowork-skills/<스킬명>/` 폴더 구조 생성 (스킬마다)

## 어조 및 스타일 규칙

- 한 번에 한 라운드만 질문한다. 여러 라운드를 한꺼번에 던지지 않는다.
- 다음 라운드로 넘어가기 전에 들은 내용을 한 문장으로 요약하여 확인한다. ("정리하면, 주요 반복 업무는 X이군요.")
- 제안 내용은 인터뷰 내용에 직접 근거한다. 일반적인 채우기 내용은 쓰지 않는다.
- 답변이 모호하면 다음 라운드로 넘어가기 전에 구체적인 예시를 한 번 더 요청한다.
- Phase 3 이전에는 어떤 파일도 생성하지 않는다.
- Phase 4는 반드시 사용자의 명시적 승인 후에만 실행한다.
- 승인 후 파일 생성은 한 번에 모두 처리하고, 중간에 추가 질문을 하지 않는다.
