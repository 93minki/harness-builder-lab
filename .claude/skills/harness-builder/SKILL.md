# Harness Builder 스킬

사용자의 실제 업무 방식과 실행 환경을 인터뷰로 수집한 뒤,
대상 플랫폼(CODEX / Claude Code / Claude Cowork)에 맞는 하네스를 설계하고,
사용자 승인 후 현재 프로젝트 디렉토리에 검증 가능한 파일을 생성하는 인터뷰어 + 생성기 스킬.

이 스킬의 목표는 "그럴듯한 초안"이 아니라 "바로 검토 가능한 실전용 하네스"다.
정보가 부족하면 생성하지 말고 추가 질문을 한다.

## 트리거

`/harness-builder` 호출 또는
"하네스 만들어줘", "내 Claude Code 환경 설정해줘", "CODEX 하네스 만들어줘",
"맞춤 스킬 구조 설계해줘" 같은 자연어 요청.

---

## 핵심 원칙

1. Phase 3 승인 전에는 어떤 파일도 생성하지 않는다.
2. 인터뷰 답변에 없는 내용을 일반론으로 채워 넣지 않는다.
3. 실행 명령, 경로, 권한, 훅, 자동화 범위는 반드시 근거를 남긴다.
4. 결정론적으로 병합할 수 없는 기존 파일은 임의 병합하지 않는다.
5. 생성 후 검증이 끝나기 전에는 "완료"라고 말하지 않는다.

---

## 전체 흐름 (6단계)

```text
Phase 0 (플랫폼 선택)
      ↓
Phase 1 (구조화 인터뷰: 역할 → 워크플로우 → 환경 → 제약 → 기존 자산)
      ↓
Phase 2 (충분성 검사 + 추가 질문 반복)
      ↓
Phase 3 (초안 제시 + CWD 확인 + 기존 파일 검사 + 병합 전략 합의)
      ↓ 승인
Phase 4 (파일 생성)
      ↓
Phase 5 (검증 + 결과 요약 + 설치/사용 안내)
```

---

## Phase 0 — 플랫폼 선택

스킬 시작 시 가장 먼저 아래 질문을 단독으로 한다.

> 어떤 플랫폼용 하네스를 만들까요?
>
> 1. **CODEX** — `AGENTS.md`, `tools/`, `docs/workflows/` 중심
> 2. **Claude Code** — `CLAUDE.md`, `.claude/settings.json`, `.claude/skills/` 중심
> 3. **Claude Cowork** — `cowork-skills/` 업로드 구조 중심
>
> 번호 또는 이름으로 선택해주세요. 두 가지 이상이면 `1+2`처럼 입력해주세요.

선택한 플랫폼만 대상으로 삼는다.
여러 플랫폼을 선택했으면 공통 목표는 유지하되,
플랫폼별 상호작용 방식과 파일 계약은 따로 설계한다.
한 플랫폼의 표현 방식을 다른 플랫폼에 억지로 맞추지 않는다.

## Phase 1 — 구조화 인터뷰

한 번에 한 라운드만 질문한다.
이전 라운드 답변을 한 문장으로 요약해 확인한 뒤 다음 라운드로 넘어간다.

답변이 모호하면 같은 라운드 안에서 예시를 요청한다.

### 라운드 1 — 역할과 성공 기준

1. 직함 / 역할이 무엇인가요?
2. 어떤 팀 또는 조직에서 일하나요?
3. 일상 업무에서 가장 중요한 목표는 무엇인가요?
4. 이 하네스가 잘 만들어졌다고 판단할 기준은 무엇인가요?

### 라운드 2 — 반복 워크플로우 인벤토리

가장 자주 하는 업무 3~5개를 각각 아래 형식으로 받는다.

- 업무 이름
- 언제 시작되는지
- 입력으로 받는 정보
- 산출물 또는 완료 조건
- 현재 수동으로 하는 단계
- 가장 자주 막히는 지점
- 실제 예시 입력 1개
- 기대하는 좋은 출력 예시 1개
- 가능하면 참고할 기존 문서/스크립트/PR 1개

추가 질문:

1. 그중 자동화 우선순위가 가장 높은 2~3개는 무엇인가요?
2. 에이전트가 처음부터 끝까지 맡아도 되는 업무와, 중간 승인이 필요한 업무를 구분해주세요.
3. 이 워크플로우에서 "좋은 결과"와 "나쁜 결과"를 짧게 비교해 주세요.

### 라운드 3 — 실행 환경과 저장소 현실

1. 주로 작업하는 OS와 셸은 무엇인가요? (예: macOS + zsh, Ubuntu + bash, Windows + PowerShell)
2. 주로 쓰는 패키지 매니저 / 런타임은 무엇인가요? (예: pnpm, npm, uv, bundler, python3)
3. 하네스가 주로 적용될 저장소 구조는 어떤가요? (예: 단일 앱, monorepo, docs-only)
4. 반복 업무에 실제로 쓰는 명령어 또는 CLI가 있나요? 있으면 예시를 주세요.
5. 자주 다루는 파일 유형이나 경로 규칙이 있나요? (예: `apps/api`, `docs/specs`, `src/`)
6. POSIX 셸, PowerShell, Python 중 실제로 사용 가능한 실행 환경을 알려주세요.
7. `make`, `just`, `bin/*`, `scripts/*`, `package.json scripts` 같은 기존 명령 래퍼가 있나요?

### 라운드 4 — 도구, 데이터, 인증

1. 주로 사용하는 툴, 서비스, 내부 시스템은 무엇인가요?
2. 연동해야 할 API, DB, 대시보드, MCP 서버가 있나요?
3. 인증이나 시크릿은 어떻게 다뤄야 하나요? 절대 파일에 넣으면 안 되는 값이 있나요?
4. 입력/출력 데이터 형식에 규칙이 있나요? (예: JSON, Markdown, SQL, CSV)
5. 생성 결과가 따라야 할 이름 규칙, 어조, 문서 형식 규칙이 있나요?

플랫폼별 추가 질문:

**Claude Code 선택 시**
5. 현재 사용 중인 `settings.json` 권한/훅/MCP 구성이 있나요?

**CODEX 선택 시**
5. CODEX에 맡길 범위는 어느 정도인가요? (단일 파일 수정 / 멀티파일 리팩터 / 기능 구현 / 조사 전용)

**Claude Cowork 선택 시**
5. Cowork에서 기대하는 상호작용 방식은 무엇인가요? (채팅, 초안 작성, 검토, 구조화된 응답 등)

### 라운드 5 — 제약사항과 안전 경계

1. 절대 자동으로 하면 안 되는 행동은 무엇인가요?
2. 보안, 컴플라이언스, 네트워크, 로깅 관련 제약이 있나요?
3. 승인 없이 허용 가능한 작업과, 항상 확인이 필요한 작업을 분리해주세요.
4. 실패 시 기본 동작은 무엇이어야 하나요? (중단, 롤백, 초안만 제시, 재시도 등)
5. 우선순위가 높은 각 워크플로우마다 아래를 짧게 적어주세요:
   - 읽기 전용 단계
   - 쓰기 단계
   - 외부 시스템 영향
   - 승인 필요 지점
   - 롤백 가능 여부

### 라운드 6 — 기존 자산과 충돌 가능성

1. 이미 있는 파일이나 규칙이 있나요? (`CLAUDE.md`, `AGENTS.md`, `.claude/settings.json`, `cowork-skills/`, `docs/templates/`)
2. 예전에 자동화를 시도했다가 잘 안 된 사례가 있나요?
3. 기존 문서/스킬/스크립트 중 반드시 재사용해야 할 자산이 있나요?
4. 반드시 따라야 할 기존 템플릿, 용어, 문체가 있나요?

### 라운드 7 — 출력 선호와 예시

1. 생성되길 원하는 스킬/태스크 이름 후보가 있나요?
2. 문서 템플릿이나 응답 형식에 선호가 있나요?
3. 대표 예시 1개를 주세요. "이 입력이 오면 이런 산출물이 나오면 좋다" 수준이면 충분합니다.
4. 가능하면 "절대 이런 식이면 안 된다"는 반례 1개도 주세요.

## Phase 2 — 충분성 검사 및 추가 질문

아래 체크리스트를 모두 충족할 때까지 Phase 3으로 넘어가지 않는다.

### 필수 체크리스트

- [ ] 역할, 팀, 핵심 목표가 명확하다
- [ ] 자동화 대상 워크플로우가 최소 3개 수집되었다
- [ ] 우선순위가 높은 워크플로우 2개 이상에 대해 입력, 단계, 산출물, 실패 지점이 파악되었다
- [ ] 우선순위가 높은 워크플로우 2개 이상에 대해 실제 예시 입력과 기대 출력 예시가 있다
- [ ] 실행 환경(OS, 셸, 런타임, 패키지 매니저)이 파악되었다
- [ ] 사용 가능한 스크립트 실행 환경(POSIX 셸 / PowerShell / Python)이 파악되었다
- [ ] 주 도구/시스템/API/MCP/DB 목록이 있다
- [ ] 안전 경계와 승인 정책이 구체적이다
- [ ] 우선순위가 높은 워크플로우마다 읽기/쓰기/외부영향/승인지점이 정리되었다
- [ ] 시크릿/인증 처리 원칙이 있다
- [ ] 기존 파일과 충돌 가능성이 확인되었다
- [ ] 재사용해야 할 기존 문서, 스크립트, 템플릿, 명령 래퍼가 확인되었다
- [ ] 성공 기준 또는 검수 기준이 있다
- [ ] 이름 규칙, 문체, 산출물 형식 규칙이 있다
- [ ] 플랫폼별로 생성할 파일 종류와 우선순위가 2개 이상 도출 가능하다

### 추가 질문 규칙

1. 부족한 항목이 있으면 해당 항목에 대해서만 1~3개 질문한다.
2. "예시를 보여달라"가 가장 빠른 경우, 예시 요청을 우선한다.
3. 추가 질문은 최대 3라운드까지 가능하다.
4. 3라운드가 지나도 핵심 정보가 비어 있으면 생성을 보류하고, 어떤 정보가 부족한지 요약한 뒤 초안 생성 대신 준비 체크리스트를 제안한다.

## Phase 3 — 초안 제시, 경로 확인, 기존 파일 검사, 병합 전략 합의

충분성 기준을 통과하면 아래 순서로 진행한다.

### 3-A. 현재 작업 디렉토리 확인

먼저 현재 환경에 맞는 명령으로 작업 경로를 확인하고 사용자에게 보여준다.

- POSIX 셸이면 `pwd`
- PowerShell 이면 `Get-Location`
- 그 외 환경이면 동등한 현재 경로 확인 명령

표시 형식:

> 현재 경로: `/path/to/project`
> 이 디렉토리에 하네스 파일을 생성합니다. 맞나요?

사용자가 다른 경로를 원하면 그 경로를 기준으로 다시 생각한다.
경로가 확정되기 전에는 초안을 제시하지 않는다.

### 3-B. 저장소 컨텍스트 탐색

기존 파일 충돌을 묻기 전에, 현재 저장소에서 재사용 가능한 자산을 먼저 확인한다.
아래 항목 중 실제로 존재하는 것만 본다.

- 명령 진입점: `package.json`, `Makefile`, `justfile`, `bin/`, `scripts/`
- 문서/규칙: `README.md`, `docs/`, 기존 템플릿, 기존 workflow 문서
- 런타임/패키지 매니저 힌트: `Gemfile`, `pyproject.toml`, `requirements.txt`, `package.json`
- CI/검증 힌트: `.github/workflows/`, `lint`/`test` 명령

탐색 목적:

- 이미 있는 명령 래퍼를 재사용한다
- 저장소에 없는 명령을 임의로 만들지 않는다
- 기존 문체, 파일 구조, 템플릿을 반영한다

탐색 결과는 초안에 반드시 요약한다.

### 3-C. 기존 파일 및 디렉토리 검사

플랫폼별로 아래 항목만 검사한다.

- Claude Code: `CLAUDE.md`, `.claude/settings.json`, `.claude/skills/`
- CODEX: `AGENTS.md`, `tools/`, `docs/workflows/`, `docs/templates/`
- Claude Cowork: `cowork-skills/`

기존 파일이 있으면 파일별로 처리 후보를 보여준다.
여기서는 절대 한 번에 전체 파일 묶음에 대해 하나의 선택을 받지 않는다.

표시 형식:

> 기존 파일이 발견되었습니다.
>
> - `CLAUDE.md`
>   1. 덮어쓰기
>   2. 결정론적 병합
>   3. 건너뛰기
>
> - `.claude/settings.json`
>   1. 덮어쓰기
>   2. 결정론적 병합
>   3. 건너뛰기

각 파일에 대해 별도로 결정을 기록한다.
특히 `.claude/settings.json`, `AGENTS.md`, 기존 스킬 파일은 개별 승인 없이는 덮어쓰지 않는다.

### 3-D. 병합 규칙

병합은 아래 규칙을 모두 만족할 때만 허용한다.

1. **Markdown 파일 (`CLAUDE.md`, `AGENTS.md`, `README.md` 일부)**
   - 기존 본문을 통째로 재작성하지 않는다.
   - 새 내용은 `<!-- harness-builder:start -->` 와 `<!-- harness-builder:end -->` 구간으로 감싼다.
   - 같은 마커 구간이 이미 있으면 그 구간만 교체한다.
   - 사용자의 기존 수기 섹션은 보존한다.

2. **JSON 파일 (`.claude/settings.json`, `skill.json`)**
   - 먼저 현재 파일을 읽고 유효한 JSON인지 확인한다.
   - 유효한 JSON일 때만 키 단위 병합을 수행한다.
   - `permissions.allow`, `permissions.deny`, `hooks` 는 안전 영향이 크므로 무조건 기계적으로 합치지 않는다.
   - 같은 키를 병합해야 하면, 사용자에게 파일별로 승인 여부를 다시 묻는다.
   - 기존 `env` 값이 비어 있지 않으면 사용자 승인 없이 덮어쓰지 않는다.
   - `.claude/settings.json` 은 사용자가 해당 파일에 대해 명시적으로 `덮어쓰기`를 고르지 않은 이상 전체 교체하지 않는다.
   - 병합 결과가 애매하면 병합하지 말고 "건너뛰기 또는 덮어쓰기"를 다시 묻는다.

3. **스킬/스크립트/템플릿 파일**
   - 동일 경로 파일이 있으면 자동 병합하지 않는다.
   - 사용자가 덮어쓰기를 선택하지 않으면 새 이름 제안을 한다.

### 3-E. 초안 제시

초안에는 반드시 아래 네 가지를 포함한다.

1. 생성 대상 파일 목록
2. 파일별 목적과 근거
3. 제안 스킬/태스크 우선순위
4. 검증 예정 항목
5. 저장소 탐색에서 재사용할 기존 자산
6. 워크플로우별 승인 경계

#### 초안 형식

```text
[선택 플랫폼]
- 생성 파일: ...
- 이 파일이 필요한 이유: ...
- 인터뷰 근거: ...
- 재사용할 기존 자산: ...

[제안 스킬/태스크]
1. 이름 / 목적 / 트리거 / 근거
2. ...

[워크플로우별 승인 경계]
- workflow A: 읽기 전용 / 쓰기 / 외부영향 / 승인 지점
- workflow B: ...

[검증 예정]
- JSON 파싱
- 경로 존재 확인
- 실행 스크립트 사용법 주석 확인
- 사용자 제약 반영 여부 확인
- 인터뷰 근거 없는 명령/경로가 없는지 확인
```

여러 플랫폼이 선택되었으면 공통 intent와 플랫폼별 차이를 구분해서 보여준다.

초안 마지막 문구:

> 이 초안대로 생성할까요?
> 수정이 필요하면 바꿀 항목을 알려주세요.

## 플랫폼별 출력 계약

아래 계약은 설명용이 아니라 생성 시 반드시 지켜야 하는 최소 규격이다.
모든 파일은 구조만 맞추면 되는 것이 아니라, 인터뷰와 저장소 탐색에서 얻은 사실에 근거해야 한다.

### 공통 근거 규칙

모든 생성 파일은 아래를 만족해야 한다.

- 인터뷰나 저장소 탐색에서 확인되지 않은 명령을 임의로 넣지 않는다
- 승인 경계는 워크플로우 단위로 구체적으로 적는다
- 파일 안의 경로, 명령, 툴 이름은 실제 사용자 환경 또는 저장소 근거가 있어야 한다
- 각 주요 섹션은 어떤 인터뷰 사실을 반영한 것인지 설명 가능해야 한다
- 구체적 근거가 부족하면 실행 스크립트 대신 문서형 산출물로 낮춘다

### CODEX 출력 계약

#### 1. `AGENTS.md`

반드시 포함:

- 하네스 목적 1문단
- 에이전트가 자주 수행할 작업 목록
- 금지 행동 / 승인 필요 행동
- 반복 스크립트와 문서의 위치
- 저장소 특화 규칙
- 작업 시작 전 확인 항목
- 실제 저장소에서 확인한 명령/경로만 사용한 작업 규칙

#### 2. `tools/<task-name>.sh` / `.ps1` / `.py`

생성 조건:

- 실제 반복 워크플로우에 대응되는 경우만 생성
- 인터뷰에서 명령어나 처리 흐름 근거가 있을 때만 생성
- `.sh` 는 POSIX 셸 사용 가능이 확인된 경우에만 생성
- `.ps1` 는 PowerShell 환경이 확인된 경우에만 생성
- `.py` 는 Python 사용 가능이 확인된 경우에만 생성

반드시 포함:

- 첫 줄 shebang
- 상단 `Usage:` 주석
- 필수 인자 설명
- 입력과 출력이 무엇인지
- 부작용과 쓰기 대상
- 실패 시 종료 코드 또는 예외 처리
- 위험 작업이면 확인 메시지 또는 안전 장치

모호하면 스크립트를 생성하지 말고 `docs/workflows/` 문서로 낮춘다.

#### 3. `docs/templates/<doc-type>.md`

반드시 포함:

- 언제 쓰는 템플릿인지
- 입력해야 할 필드
- 예시 섹션 제목

#### 4. `docs/workflows/<workflow>.md`

반드시 포함:

- 시작 조건
- 필요한 입력
- 단계별 절차
- 사람 승인 지점
- 실패 시 대응
- 구체적 명령 또는 문서 경로 근거

### Claude Code 출력 계약

#### 1. `CLAUDE.md`

반드시 포함:

- 프로젝트/업무 컨텍스트
- 자주 쓰는 명령어 또는 작업 패턴
- 승인 정책
- 금지 행동
- 생성된 스킬 목록과 사용 시점
- 관련 문서/템플릿 경로
- 저장소 탐색에서 재사용한 명령/문서 자산

#### 2. `.claude/settings.json`

생성 조건:

- 인터뷰에서 실제로 필요한 권한, 훅, 환경변수가 파악된 경우만 생성 또는 수정

반드시 지킬 규칙:

- 유효한 JSON만 쓴다
- `...` 같은 placeholder를 남기지 않는다
- `allow` 에 광범위 와일드카드를 넣지 않는다
- hook 명령은 구체적인 목적과 트리거가 있을 때만 추가한다
- 시크릿은 값으로 직접 쓰지 않는다
- 기존 `permissions` 와 `hooks` 는 파일별 승인 없이 자동 병합하지 않는다

최소 형식 예시:

```json
{
  "permissions": {
    "allow": [],
    "deny": []
  }
}
```

추가한 각 키에 대해 초안 또는 완료 요약에서 이유를 설명한다.

#### 3. `.claude/skills/<skill-name>/SKILL.md`

반드시 포함:

- YAML frontmatter: `name`, `version`, `description`
- 언제 써야 하는지
- 언제 쓰면 안 되는지
- 트리거
- 목적
- 입력
- 단계
- 출력
- 제약
- 실패 또는 에스컬레이션 조건
- 좋은 출력과 피해야 할 출력의 기준

생성 조건:

- 실제 반복 업무가 있고,
- 트리거 문구와 성공 산출물이 명확할 때만 생성

#### 4. `docs/templates/`

문서 기반 워크플로우가 실제로 존재할 때만 생성한다.

### Claude Cowork 출력 계약

#### 1. `cowork-skills/<skill-name>/skill.json`

반드시 유효한 JSON이어야 하며 아래 키만 기본으로 사용한다.

{
  "name": "string",

  "trigger": "string",

- 금지 행동
- 출력 형식
- 승인 요청이 필요한 경우의 표현 방식

#### 3. `cowork-skills/<skill-name>/README.md`

반드시 포함:

- 스킬 설명
- 트리거
- 예시 사용법
- 제약사항

## 기본 생성 규칙

1. 선택한 플랫폼 파일만 생성한다.
2. 필요한 부모 디렉토리가 없으면 먼저 생성한다.
3. 기존 파일 처리 방침을 어기지 않는다.
4. 인터뷰 근거가 약한 스크립트/훅/권한은 생성하지 않는다.
5. 같은 목적의 산출물이 문서형으로 충분하면 실행 스크립트 생성을 강행하지 않는다.
6. 멀티플랫폼 생성 시 공통 제약은 공유하되, 플랫폼별 표현과 상호작용 방식은 따로 최적화한다.
7. 저장소 탐색에서 찾은 기존 명령 래퍼와 문서를 우선 재사용한다.
8. 사용자 환경이 불명확하면 실행 가능한 스크립트보다 문서형 workflow를 우선한다.

## Phase 4 — 파일 생성

사용자가 초안을 명시적으로 승인하면 생성한다.

### 생성 순서

1. 필요한 디렉토리를 현재 환경에 맞는 방식으로 먼저 생성한다
   - POSIX 셸이면 `mkdir -p`
   - PowerShell 이면 `New-Item -ItemType Directory -Force`
2. 기존 파일 처리 방침(덮어쓰기/병합/건너뛰기)을 적용한다
3. 플랫폼별 파일을 작성한다
4. 생성 직후 검증을 수행한다

생성 중에는 추가 질문을 하지 않는다.
추가 질문이 필요해질 정도로 정보가 부족했다면, 원래부터 생성하면 안 되는 상태다.

## Phase 5 — 검증 및 완료 보고

파일 생성 후 반드시 아래 검증을 수행한다.

### 공통 검증

- 생성된 모든 파일 경로가 존재하는지 확인
- 선택하지 않은 플랫폼 파일이 생기지 않았는지 확인
- 사용자 제약과 금지 행동이 산출물에 반영되었는지 확인
- 주요 명령, 경로, 툴 이름이 인터뷰나 저장소 탐색 근거를 가지는지 확인

### 형식 검증

- JSON 파일은 파싱 가능한지 확인
- Markdown 파일은 비어 있지 않은지 확인
- 스크립트 파일은 `Usage:` 주석과 실행 진입점이 있는지 확인
- 남아 있는 placeholder (`<...>`, `TODO`, 의미 없는 `...`)가 없는지 확인

### 실행 가능성 검증

- 셸 스크립트가 있으면 `sh -n` 또는 `bash -n` 으로 문법 확인
- PowerShell 스크립트가 있으면 `pwsh -NoProfile -Command "[scriptblock]::Create((Get-Content -Raw '<file>')) | Out-Null"` 또는 동등한 구문 검사로 확인
- Python 스크립트가 있으면 `python -m py_compile` 로 문법 확인
- 파일 내에서 언급한 경로, 스킬명, 템플릿명이 실제로 존재하는지 확인
- POSIX 환경이 아닌데 `.sh` 가 생성되지는 않았는지 확인
- PowerShell 환경이 아닌데 `.ps1` 이 생성되지는 않았는지 확인
- Python 사용 불가 환경인데 `.py` 가 생성되지는 않았는지 확인

### 일관성 검증

- 초안에 약속한 파일과 실제 생성 파일이 일치하는지 확인
- 멀티플랫폼일 때 공통 intent와 승인 정책이 서로 충돌하지 않는지 확인
- `CLAUDE.md`, `AGENTS.md`, `prompt.md` 간 금지 행동이 모순되지 않는지 확인
- 각 우선순위 workflow 에 대해 입력, 출력, 승인 경계가 문서에 모두 반영되었는지 확인
- 구조만 맞고 내용이 generic 하지 않은지, 인터뷰 예시와 실제 저장소 자산을 반영했는지 확인

### 검증 실패 시 규칙

1. 가능한 경우 즉시 수정한다
2. 자동 수정이 위험하면 파일을 남기되 실패 항목을 명시한다
3. 검증 실패 상태를 숨기지 않는다

### 완료 보고 형식

```text
생성 완료

생성된 파일:
- ...

검증 결과:
- 실행한 검사: ...
- 통과: ...
- 주의: ...

다음 단계:
- ...
```

검증 실패가 남아 있으면 아래 형식을 쓴다.

```text
검증 실패

실패한 검사:
- ...

영향받는 파일:
- ...

다음 조치:
- ...
```

설치/사용 안내는 선택된 플랫폼에 대해서만 보여준다.
"자동 적용된다" 같은 단정 표현보다, 사용자가 검토해야 할 지점을 함께 알려준다.

## 어조 및 스타일 규칙

- 한 번에 한 라운드만 질문한다
- 다음 라운드 전에 들은 내용을 한 문장으로 요약한다
- 일반론적 채우기 문장을 피한다
- 답변이 모호하면 예시를 다시 요청한다
- 승인 전에는 어떤 파일도 생성하지 않는다
- 검증 전에는 완료 선언을 하지 않는다
